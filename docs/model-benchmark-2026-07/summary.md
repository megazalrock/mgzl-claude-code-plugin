# review:diff レビュアーモデル比較ベンチマーク（2026-07）

`review:diff` スキルのレビュアー起動モデル切り替え閾値（f2adb19 で導入、2f33c77 で 100行→50行に引き下げ）に
根拠がなかったため、sonnet / opus の検出力差を実測した。

## 方式

- 既知のバグ・欠陥を埋め込んだ TypeScript フィクスチャ（v0=変更前 / v1=変更後）から `git diff --no-index` で差分を生成
- `reviewer-for-logic` / `reviewer-for-comments` のエージェント定義を読み込ませた汎用サブエージェントに、
  同一プロンプト・同一入力で `model` パラメータのみ sonnet / opus に変えてレビューさせる
- レビュー結果を正解データ（ground-truth*.md）と突き合わせ、検出率・誤検出・severity 較正を採点
- 全3ラウンド・計18run。ラウンド1は素直なバグ（JSDoc矛盾のヒントあり）、ラウンド2・3は
  推論しないと見つからない hard バグ（ヒントなし）

## 帯域別の検出成績（sonnet / opus）

| 帯域 | easy (R1) | hard (R2/R3) |
|---|---|---|
| <50行 | 3/3 = 3/3 | logic 4/4 = 4/4、comments **5/5 > 4/5** |
| 50〜99行 | 5/5 = 5/5 | logic **4/5 < 5/5**、comments **6/7 < 7/7** |
| ≥100行 | 5/6+部分 < 6/6 | logic 5/6 < 6/6、comments 9/9 = 9/9 |

## 結論

1. **現行の50行閾値は帯域別成績と正確に整合する**
   - sonnet の完全な見逃し（R2: HM4 浅いコピー汚染・CM2 指数/線形バックオフ矛盾、R1: L1 独立指摘漏れ）は
     **すべて50行以上**で発生
   - opus の唯一の見逃し（R3: CS5 用語不統一。当該コメントを「良い点」として賞賛までした）は
     **50行未満**で発生
   - 50行未満では hard バグでも sonnet は互角以上 → sonnet 割り当てで品質低下の証拠なし
2. 差を生む本質は行数ではなく**バグの微妙さ**（行数は難易度の弱い代理変数）。
   素直なバグなら 74 行でも sonnet は完璧だった
3. opus は 18run 中 16run で省トークン（約2〜3割減）かつ実行時間も同等以下。
   opus が劣るのはトークン単価のみ
4. severity 較正は opus が一貫して優秀。sonnet は小型差分で [5] に過大評価する傾向
5. 誤検出は両モデルともほぼゼロ。「指摘してはいけない正当コメント」も正しくスルー
   （opus が正解データ上「正当」の RFC 4180 コメントを指摘した1件は、理屈が事実として正しく境界例扱い）

## 限界

- 各条件 n=1 で分散未測定（特に R3 の sonnet 勝利は分散の産物の可能性あり）
- 単一ファイル差分のみ。ファイル横断の文脈が必要なバグは未検証
- 正解データが自作（作問者=採点者のバイアス）

## ディレクトリ構成

- `summary.md` — 本ファイル
- `scoring-summary.md` — ラウンドごとの詳細採点（見逃しの内訳・実行時間・トークン）
- `ground-truth.md` / `ground-truth-round2.md` / `ground-truth-round3.md` — 埋め込みバグの正解データ
- `v0/` / `v1/` — フィクスチャ（変更前/変更後）。再実行時はこの2つから diff を再生成する
- `diffs/` — レビュアーに渡した差分ファイル
- `results/` — 全18run のレビューレポート原文（`<fixture>-<model>.md`）
