# コメントレビュー結果（reviewer-for-comments）

## member-cache.ts

### ✅ 良い点

- `displayNameOf` の `// ユーザーの表示名が未設定の場合はメールアドレスのローカル部を使う` は、`member.email.split('@')[0]` が「ローカル部」を取り出す意図（フォールバック動作）を的確に説明できており、実装と一致しています。短く読みやすく、良いコメントです。

### [4] 強く推奨

**問題**: `// キャッシュは 5 分で失効する` が定数の値と矛盾しています。直下の定義は `const CACHE_TTL_MS = 600_000` で、これは 600,000 ミリ秒 = 600 秒 = **10 分**です。コメントの「5 分」は実装と一致しません。
**理由**: 読み手はコメントを信じて「5 分でキャッシュが切れる」前提の呼び出しコードや運用判断をしてしまいます。値と説明が食い違うコメントは誤誘導になります。
**提案**:
```typescript
// キャッシュは 10 分で失効する
const CACHE_TTL_MS = 600_000
```
（もし本来 5 分が正なら値側の修正が必要ですが、それは実装の是非としてロジックレビュー範囲です。ここではコメントと現行値の不一致のみを指摘します。）

---

**問題**: `getMember` の JSDoc `@throws 会員 ID の形式が不正な場合はエラーを送出する` が、実際の挙動と正反対です。該当分岐は例外を送出せず `return null` しています。
```typescript
if (!/^m_[0-9a-z]{8}$/.test(memberId)) {
  return null
}
```
**理由**: 「形式不正時に throw する」と書かれていると、呼び出し側は `try/catch` で不正 ID を捕捉しようとし、実際には `null` が返るため分岐を取りこぼします。実挙動と反対を述べる誤誘導コメントで、単なる陳腐化より有害です。
**提案**:
```typescript
/**
 * 会員をキャッシュ経由で取得する。
 * @returns 会員 ID の形式が不正な場合は null を返す
 */
```
（ネットワーク層で `axios` が投げうる例外を明示したい場合は、別途 `@throws` を実挙動に合わせて記載してください。）

**問題**: `hasSameEmail` の `// メールアドレスは大文字小文字を区別せずに比較する` が実装と一致しません。コメントは「大文字小文字を区別しない比較」と宣言していますが、コードは一方（`a.email`）しか小文字化していません。
```typescript
return a.email.toLowerCase() === b.email
```
**理由**: 「case-insensitive で比較する」というコメントを信じた読み手は、`b.email` に大文字が含まれても一致すると誤解します。実際には `b` 側は正規化されておらず、宣言どおりの比較になっていません。コメントの主張が実装の動作を上回っている誤誘導です（実装側の修正要否はロジックレビュー範囲のため、ここではコメントと実装の乖離のみを指摘します）。
**提案**: コメントを実装の実挙動に合わせるか、実装をコメントに合わせて両辺を正規化するかのいずれかで整合させてください。コメント側で正確を期すなら、たとえば次のように現状を正しく説明します。
```typescript
// a のメールアドレスのみ小文字化して b と比較する
```

---

**問題**: `buildMemberLabel` 直上の `// TODO: v2 API 移行後にこの分岐を削除する` が、存在しないコード構造を指しています。当該関数はテンプレートリテラルを返すだけで、削除対象となる「分岐（if 等）」がありません。
```typescript
export const buildMemberLabel = (member: Member): string => {
  return `${member.displayName} <${member.email}>`
}
```
**理由**: 「この分岐を削除する」という TODO は、対象となる分岐が実在してこそ意味を持ちます。分岐が無い関数に付いているため、読み手はどのコードを消せばよいか判断できず、TODO が死んでいる（対象を失っている）状態です。
**提案**: TODO の対象を実在するコードに合わせるか、対象が消えているなら TODO ごと削除してください。関数自体を将来置き換える意図なら、対象を明確にします。
```typescript
// TODO: v2 API 移行後に buildMemberLabel を廃止する
```

## 📚 参考情報

- JSDoc の `@throws` / `@returns` は実際のシグネチャ・制御フローと一致させることが原則です（例外を投げるのか、値で返すのかを取り違えない）。
- TODO / FIXME は「何を・いつ・どうする」を実在するコードに紐づけて記述し、対象が消えたら同時に除去すると陳腐化を防げます。
