### [4] 強く推奨

**問題**: 9行目のコメント「// キャッシュは 5 分で失効する」が、直後の `const CACHE_TTL_MS = 600_000` の実際の値と一致しない。600,000ms は 10分であり、5分ではない。
**理由**: 保守者がこのコメントを信頼してTTLの挙動を誤認する可能性がある。実装値とコメントの数値が食い違う典型的な不整合であり、優先して直すべき。
**提案**:
```typescript
// キャッシュは 10 分で失効する
const CACHE_TTL_MS = 600_000
```

**問題**: `getMember`（19-22行目）の JSDoc は `@throws 会員 ID の形式が不正な場合はエラーを送出する` としているが、実装（24-26行目）では ID 形式が不正な場合は例外を送出せず `null` を返して早期リターンしている。実際に例外が発生し得るのは、キャッシュが無効で `axios.get` を呼ぶ経路（31行目）でネットワーク/APIエラーが起きた場合である。加えて `@param memberId` と `@returns` のタグも欠落しており、シグネチャと対応していない。
**理由**: 「不正な ID を渡すと throw する」という誤った前提で呼び出し側が書かれると、null チェック漏れや不要な try/catch 実装など、ドキュメントの実態と正反対の内容を信じたことによるバグを誘発する。misleadingなコメントは単なる陳腐化より深刻。
**提案**:
```typescript
/**
 * 会員をキャッシュ経由で取得する。
 * @param memberId - 会員 ID
 * @returns 会員 ID の形式が不正な場合は null を返す。キャッシュが有効なら
 *   キャッシュ済みの会員を、無効なら API から取得した会員を返す
 * @throws API 呼び出し（axios.get）が失敗した場合にエラーを送出する
 */
```

**問題**: 37行目のコメント「// メールアドレスは大文字小文字を区別せずに比較する」に対し、実装は `a.email.toLowerCase() === b.email` であり、`b.email` は小文字化されていない。`b.email` に大文字が含まれる場合、実際には大文字小文字を区別しない比較になっていない。
**理由**: コメントを信じた呼び出し側は、大文字小文字の違いがあるメールアドレス同士でも一致すると誤解し得る。コメントの主張（両辺とも大文字小文字を無視）と実装（片側のみ小文字化）が食い違っている。
**提案**: コメントを実態に合わせて修正するか（下記例）、実装側を `b.email.toLowerCase()` に揃えて本来のコメント通りの挙動にするか（実装修正の要否はロジックレビュー側の判断）を明確にする。
```typescript
// 注意: a.email のみ小文字化しており、大文字小文字を厳密に無視した比較にはなっていない
export const hasSameEmail = (a: Member, b: Member): boolean => {
  return a.email.toLowerCase() === b.email
}
```
