# コメントレビュー結果（reviewer-for-comments）

## retry-client.ts

### ✅ 良い点

- `postOnce` の JSDoc（「冪等でない POST は再試行しない。呼び出し側でリトライの可否を判断できるよう、エラーはそのまま伝播させる。」）は、コードが何をするか（`what`）ではなく、なぜそう実装しているか（`why`）を説明できており、実装（try/catch を持たず素通しでエラーを伝播）とも一致しています。良いコメントです。

### [4] 強く推奨

#### 4-1. `fetchWithRetry` の JSDoc「見つからない場合は null を返す」が実装と矛盾している

**問題**: JSDoc（20行目付近）に「リソースが見つからない場合は null を返す。」とありますが、実装では 404 を検知すると `throw new NotFoundError(...)`（37行目）で**例外を送出**しており、`null` は一切返していません。
**理由**: コメントが実際の挙動と正反対のことを述べているミスリーディングなコメントです。これを信じた呼び出し側は「戻り値が null かどうか」で分岐するコードを書いてしまい、実際には投げられる `NotFoundError` を捕捉できずに落ちます。
**提案**:
```typescript
/**
 * リソースを取得する。失敗時は指数バックオフで再試行する。
 * リソースが見つからない場合（404）は NotFoundError を送出する。
 */
```

#### 4-2. `@param options.timeoutMs` が存在しないフィールドを指しており、単位も誤っている

**問題**: JSDoc（23行目）に `@param options.timeoutMs リクエスト単体のタイムアウト（ミリ秒）` とありますが、`RetryOptions` に `timeoutMs` というフィールドは存在せず、実際のフィールドは `timeoutSeconds`（6行目）です。実装も `options.timeoutSeconds * 1000`（32行目）と**秒**を前提にしています。コメントの「ミリ秒」も誤りです。あわせて、`options.maxAttempts` と `options.baseDelayMs` は `@param` が欠落しています。
**理由**: 参照しているシンボル名（`timeoutMs`）がリポジトリ内に存在せず、単位の記述もフィールドの意味（秒）と食い違っています。読者が「ミリ秒で渡す」と誤解し、1000倍のタイムアウト値を設定する事故につながります。
**提案**:
```typescript
/**
 * @param url 取得先の URL
 * @param options.maxAttempts 最大試行回数
 * @param options.baseDelayMs バックオフの基準遅延（ミリ秒）
 * @param options.timeoutSeconds リクエスト単体のタイムアウト（秒）
 */
```

#### 4-3. JSDoc の「指数バックオフ」がコードの線形バックオフと一致しない

**問題**: JSDoc（19行目）は「指数バックオフで再試行し」と述べていますが、実装の遅延計算は `const delay = options.baseDelayMs * attempt`（49行目）で、試行回数に**線形**比例する遅延です。指数バックオフ（例: `baseDelayMs * 2 ** attempt`）ではありません。
**理由**: コメントが記述するアルゴリズムと実装のアルゴリズムが食い違っています（実装の是非ではなく、コメントと実装の整合性の問題として報告します）。読者はコメントを根拠に遅延の増加を過大に見積もります。
**提案**: 実装が線形のままなら「試行回数に比例した遅延（線形バックオフ）で再試行し」と修正する。指数バックオフが意図なら実装側の修正が必要ですが、それはロジック担当レビュアーの領域です。

#### 4-4. `// Retry-After ヘッダの解釈は parseRetryAfter() に委ねる` が存在しない関数を参照し、挙動とも矛盾

**問題**: 48行目のコメントは `parseRetryAfter()` に処理を委ねると述べていますが、この関数はファイル内に定義・インポートされていません（対象ファイル全体を確認済み）。また直後の遅延計算は `options.baseDelayMs * attempt`（49行目）のみで、`Retry-After` ヘッダの解釈は一切行われていません。
**理由**: 参照先シンボルがリポジトリ内に実在せず、かつコメントが述べる挙動（Retry-After に基づく遅延）が実装に存在しません。存在しない委譲を示唆するコメントは読者を強く誤導します。
**提案**: `Retry-After` を実際に扱っていないため、コメントを削除する。将来対応するなら TODO として残す。
```typescript
// TODO: Retry-After ヘッダに基づく遅延調整は未対応
const delay = options.baseDelayMs * attempt
```

### [3] 推奨

#### 3-1. `RETRYABLE_STATUS` の説明コメントが冗長・迂遠

**問題**: 11行目「リトライ対象とするステータスコードに関しては、サーバー側の一時的な過負荷という形で発生するものを対象とするという方針で選定を行っている」は約60字の一文で、「〜に関しては」「〜という形で」「〜という方針で」という迂遠な言い回し（criterion 5 の circumlocution）が重なっています。
**理由**: 一文が長く、冗長な連結表現のために趣旨が読み取りにくくなっています。
**提案**: 「一時的な過負荷を示すステータスコードのみをリトライ対象とする。」

#### 3-2. `// カウンタをインクリメント` は自明な処理の言い換え

**問題**: 43行目の `// カウンタをインクリメント` は、直後の `attempt += 1` が自明に表している内容をそのまま言い換えているだけです。
**理由**: コードを読めば分かる `what` を繰り返しており、コメントとしての価値（rent）を持ちません。
**提案**: 削除する。

#### 3-3. コメントアウトされたコードが残存

**問題**: 50行目の `// const legacyDelay = 3000` は、旧実装と思われるコメントアウト済みコードです。
**理由**: 削除された実装は Git 履歴に残るため、コメントアウトされたコードはファイルに残す価値がほとんどありません。
**提案**: 削除する。
