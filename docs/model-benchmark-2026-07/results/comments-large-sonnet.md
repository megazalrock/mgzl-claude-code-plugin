# コメントレビュー結果（reviewer-for-comments）

## src/export-pipeline.ts

### ✅ 良い点

- `formatCsvRow` 上の `// RFC 4180 に準拠し、カンマまたはダブルクォートを含む値のみクォートする` は、実装（カンマまたは `"` を含む場合のみクォートする分岐）と一致しており、非自明な仕様判断を簡潔に説明できています。
- `JOB_ID_PATTERN` 上の `// ジョブ ID はサーバー側で UUID v4 として発行される` は、正規表現中のバージョン桁 `4` やバリアント桁 `[89ab]` がなぜその値なのかを補足する良い「why」コメントです。

### [4] 強く推奨

1. **問題**: `POLL_INTERVAL` の `/** ポーリング間隔（ミリ秒） */` は、実際の使用箇所 `setTimeout(resolve, POLL_INTERVAL * 1000)` と矛盾しています。`* 1000` を掛けて `setTimeout`（ミリ秒指定）に渡している以上、`POLL_INTERVAL = 5` は「秒」であり「ミリ秒」ではありません。
   **理由**: コメントを信じた読者がこの定数をミリ秒として別の API にそのまま渡すと、待機時間が 1000 分の 1 になってしまう実害のある誤解を招きます。
   **提案**:
   ```typescript
   /** ポーリング間隔（秒） */
   const POLL_INTERVAL = 5
   ```

2. **問題**: `selectExportTargets` の Doc コメントは「アーカイブ済みのレコードは対象から除外する」と書かれていますが、実装は `records.filter((r) => r.archived)` であり、`archived` が `true` のレコード（＝アーカイブ済みのレコード）のみを残しています。これはコメントの主張と正反対の挙動です。
   **理由**: 実装と真逆の説明をするコメントは、呼び出し側の実装ミスを誘発する典型的な誤誘導コメントです。
   **提案**: 現在の実装に合わせるなら次のように修正してください（もし「非アーカイブのレコードを対象にする」が本来の意図であれば、それはロジック側の確認が必要なため、コメント修正とあわせて実装担当者に確認することを推奨します）。
   ```typescript
   /**
    * エクスポート対象のレコードを返す。
    * アーカイブ済みのレコードのみを対象とする。
    */
   ```

3. **問題**: `sortByCreatedAt` の説明文と `@returns` タグの両方に「昇順」と記載されていますが、比較関数は `b.createdAt.localeCompare(a.createdAt)` であり、これは降順（新しい日時が先）にソートします。
   **理由**: 説明・タグの双方が実際の並び順と正反対であり、呼び出し側が並び順を誤認するリスクが高いです。
   **提案**:
   ```typescript
   /**
    * レコードを作成日時の降順で並べ替えて返す。
    * @returns 作成日時の降順でソートされた新しい配列
    */
   ```

4. **問題**: `exportToJson` の Doc コメントは「レコードを CSV 形式でエクスポートする」「ヘッダ行を先頭に付与し、各レコードをカンマ区切りの行として出力する」と説明していますが、実装は `JSON.stringify(...)` で JSON 文字列を返しており、CSV とは無関係です。`toCsv` のコメントをそのまま使い回した際の貼り付けミスと見られます。
   **理由**: フォーマットの説明が完全に事実と異なり、戻り値の形式を誤解させる典型的な誤誘導コメントです。
   **提案**:
   ```typescript
   /**
    * レコードを JSON 形式でエクスポートする。
    * エクスポート日時（exportedAt）を付与したオブジェクトを、整形済みの JSON 文字列として返す。
    */
   ```

5. **問題**: `toCsv` 直前の `// CSV のエスケープ処理は後述の buildCsvRow() で行う` は、存在しない関数名 `buildCsvRow` を参照しています。実際にエスケープ処理を行っているのは直後に定義されている `formatCsvRow` です。
   **理由**: 参照先のシンボル名が実装と一致しておらず、読者がコードを探しても該当箇所を見つけられません。
   **提案**:
   ```typescript
   // CSV のエスケープ処理は後述の formatCsvRow() で行う
   ```

6. **問題**: `startExportJob` 内の `// レビュー指摘対応: LOGIC-3` は、修正の経緯（レビュー対応履歴）を記録したコメントであり、コード自体の説明になっていません。
   **理由**: レビュー対応履歴は Git 履歴や PR の会話に残すべき情報であり、ソースコード中に残すとコードの可読性を下げます。
   **提案**: コメントごと削除してください。
   ```typescript
   export const startExportJob = async (recordIds: string[]): Promise<ExportJob> => {
     const res = await axios.post(`${API_BASE}/exports`, { recordIds })
     return res.data
   }
   ```

### [3] 推奨

1. **問題**: `summarize` 内の `// 結果を返す` は `return { count: records.length, totalSizeBytes }` の直前にありますが、コードを読めば自明な内容をそのまま言い換えているだけです。
   **理由**: 実装から自明に読み取れる内容であり、コメントとしての付加価値がありません。
   **提案**: コメントを削除してください。

2. **問題**: `waitForJob` 内の `// 🚀 fast path: 終端状態なら即 return` に絵文字 `🚀` が含まれています。
   **理由**: 装飾的な絵文字はコードコメントとしてノイズになり、意味はテキストのみで伝えるべきです。
   **提案**:
   ```typescript
   // fast path: 終端状態なら即 return
   ```

3. **問題**: `runExport` の Doc コメント内で丸数字 `①②③` が使用されています。
   **理由**: 丸数字は環境によって表示崩れや検索性の低下を招くため、通常の番号表記を使うべきです。
   **提案**:
   ```typescript
   /**
    * エクスポートの流れ:
    * 1. 対象レコードの絞り込み
    * 2. ジョブの開始
    * 3. 完了までポーリング
    */
   ```

### [1] 情報

- `formatCsvRow` の `// RFC 4180 に準拠し...` は外部仕様（RFC）への言及です。RFC 番号自体は安定した識別子で曖昧さは少ないため必須ではありませんが、余裕があれば `https://www.rfc-editor.org/rfc/rfc4180` のような URL を添えるとより辿りやすくなります。
