CBOプラグインで、レビュー結果をDBに保存しレビューの精度の記録を取っていく仕組みを作りたいと思います。

- summary.md このファイル
- spec.md 詳細な使用
- tasks.md 実装タスク

## 用語

- レビュー報告書（report）： 1度のレビューで出力されたレビュー報告書のファイル、findingが複数含まれる
- 指摘内容（finding）： レビュー報告書内の指摘内容、
- 評価（verdict）： tp/fp/nit（true_positive、false_positive、nitpick）に分類される人間やAIがつける「本当に修正すべきだったか」の評価
  - tp：指摘内容が正しい（true positive）
  - fp： 指摘内容が間違っている（false positive）
  - nit：指摘内容が些末、品質向上に寄与しない（nitpick）
  - oos：指摘内容は正しいが、スコープ外、修正を見送り（out of scope）

## ゴール

1. `/review:diff` やGitHubCopilot、Codex、`/code-review` などから作成したレビュー報告書元に指摘内容をDBに記録し、本当に修正が必要だったか（偽陽性） を記録し、今後のAIによるレビュー精度の向上のための情報を収集する
2. ClaudeCodeが利用できる知識として抽出する（次のタスク）

## 記録する内容

### レビュー報告書（report）
- レビュー報告書ID（UUIDv7）
- レビュー報告書のファイルパス（メタデータ）
- レビュー報告書の作成日時
- 使用したモデル名（Codex / Copilot の場合は不明な可能性あり。なるべくレビュー報告書側に含めるように工夫する）

### 指摘内容（finding）
- 指摘内容ID（UUIDv7、グローバル一意）
- 紐づくレビュー報告書ID
- 指摘内容本文（全文）
- 指摘箇所
  - ファイルパス
  - 行番号（レンジ）
  - 抜粋コード（before / after）
- 指摘内容の重要度（5〜1の5段階）
  - 5：必須修正 (ブロッカー)
  - 4：強く推奨
  - 3：推奨 （基本的に3以上が修正対応）
  - 2：軽微
  - 1：情報
- 指摘内容の評価：tp/fp/nit/oos、空（=未評価）を許容する
- 指摘内容の種別：design/logic/style/comments/security/performance/test
- 指摘内容の報告者：サブエージェントの場合はサブエージェント名、その他の場合は `Codex` `ClaudeCode code-review` `GitHub Copilot` または任意の文字列（当てはまらない場合もあとから識別できるようにする）
- 評価の根拠コメント（nullable）

### 含めない項目
- PR番号 / ブランチ名 / 対象コミットSHA
- 人間による評価者情報（評価は人間が一次的に行うが、個人利用前提のため評価者の識別情報は持たない）
- 評価日時（レビュー報告書の作成日時で代用）
- 重複指摘リンク（将来必要になれば別テーブルで導入）
- 修正コミットSHA

## 方針（検討事項の決定）

| 検討事項 | 決定 |
|---|---|
| 指摘内容本文を要約とするか全文とするか？ | **全文＋構造化フィールド**（ファイルパス・行番号・抜粋コード等を別カラムに分離） |
| レビュー報告書IDをどうするか？ | **UUIDv7 を発番**。ファイルパスはメタデータとして別カラムに保持 |
| 指摘内容のID体系をどうするか？ | **UUIDv7**（グローバル一意） |
| ClaudeCode からのDBアクセスはどうするか？ | **bun スクリプト経由（SQLite）**。プラグイン配布性と既存ルール（bun+TS）との整合性を優先。SQLite ドライバは Bun 組み込みの `bun:sqlite`（`drizzle-orm/bun-sqlite`）を採用（`better-sqlite3` は Bun 1.3 系の Node ABI と非互換のため不採用） |
| ClaudeCode からの操作 UX は？ | **`review:import-report` スキルで CLI をラップ**。「レビュー報告書をインポートして」等の発話で起動。スクリプトは `cbo/skills/review__import-report/scripts/review-db.ts` |
| 評価は誰が行うか？ | **人間がレビュー報告書ファイルに直接書き込む**。AI による評価は本スコープ外（必要時にユーザーが Claude Code に直接指示する想定） |
| レポートのメタデータをどう持つか？ | **レビュー報告書ファイル先頭の YAML フロントマター**（`reporter` / `model` 必須、任意項目は記録メモのみで DB には保存しない）。`report.created_at` はファイルの mtime |
| CLI 引数パーサは？ | **`util.parseArgs`（Node.js 標準・bun 互換）**。外部ライブラリは導入しない |
| 抽出した知識はどうするか？ | **`knowledge__distill` スキルを起点とした蒸留パイプライン化**（※ 次回タスクで扱う） |
