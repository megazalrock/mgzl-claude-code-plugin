# Mutation Testing

ソースコードにミューテーション（意図的な小変更）を適用し、テストスイートがそれを検出できるかを
検証することで、テストの品質を計測する。
worktree 隔離環境で安全にミューテーションを実行する。

## 使い方

```
/mutation-testing                          # git diff main...HEAD の変更ファイルを対象
/mutation-testing src/game/score.ts        # 特定ファイルを対象
/mutation-testing src/foo.ts src/bar.ts    # 複数ファイル指定
/mutation-testing all                      # src/ 配下の全ソースファイルを対象
```

## 実行フロー

### Step 1: テストランナーの検出

プロジェクトの `package.json` を読み取り、使用しているテストランナーを特定する:

1. `package.json` の `scripts.test` を確認
2. 依存関係から判定: `bun` → `bun test`, `vitest` → `vitest`, `jest` → `jest`
3. 検出できない場合は AskUserQuestion でユーザーに確認する

### Step 2: 対象ファイルの特定

引数 `$ARGUMENTS` に応じて対象ファイルを決定する:

**引数なしの場合**:
```bash
git diff main...HEAD --name-only --diff-filter=ACMR
```
結果から `.ts` / `.js` / `.tsx` / `.jsx` ファイルのみ抽出し、テストファイル（`__tests__/`, `.test.ts`, `.spec.ts`）と型定義ファイル（`.d.ts`）を除外する。

**特定ファイル指定の場合**:
`$ARGUMENTS` で指定されたファイルパスをそのまま使用する。ファイルの存在を確認すること。

**`all` の場合**:
`src/` 配下の全ソースファイルを検索する。テストファイル、型定義ファイル、`index.ts` は除外する。
```bash
find src -name "*.ts" -o -name "*.tsx" | grep -v -E '(__tests__|\.test\.|\.spec\.|\.d\.ts|index\.ts)'
```

対象が0件の場合は「ミューテーション対象のソースファイルがありません」と報告して終了する。

対象ファイル一覧をユーザーに報告する:
```
ミューテーションテスト対象:
- src/utils/calc.ts
- src/domain/validator.ts
```

### Step 3: エージェントの実行

`@mutation-tester` エージェントを **`isolation: worktree`** で起動する。
対象ファイルが複数ある場合は、ファイルごとにサブエージェントを並列で起動する。

各エージェントへの指示:
```
以下のソースファイルに対してミューテーションテストを実行してください。

## テストランナー
{検出したテストランナーコマンド}

## 対象ファイル
{対象ファイル（1ファイル）}

ミューテーション計画を立て、1つずつ適用・テスト・記録・復元のサイクルを実行し、
レポートと最終サマリーを出力してください。
```

### Step 3.1: エージェント実行失敗時の処理

エージェントが worktree/sandbox 関連のエラーで中止した場合（「Operation not permitted」、コマンド実行不可、依存関係の問題等）:

1. **Step 4（統合レポート）をスキップ**する
2. エージェントから報告されたエラー内容をそのままユーザーに伝える:
   - 失敗したコマンド・操作
   - エラーメッセージ
   - 推定原因
3. 推奨対処を提示する:
   - sandbox 設定の確認（`/sandbox` コマンド）
   - worktree 対象ディレクトリの書き込み権限
   - 依存関係のインストール状態

**エージェントが正常に完了した場合のみ** Step 4 に進む。

### Step 4: 統合レポートの出力

エージェントの結果を受け取り、以下の形式で統合レポートを出力する:

```markdown
# Mutation Testing Report

## サマリー
- 対象ファイル数: {N}
- 総ミューテーション数: {total}
- KILLED (検出): {killed} ({killed_pct}%)
- SURVIVED (未検出): {survived} ({survived_pct}%)
- 総合スコア: {score}% ({rating})

## ファイル別スコア

| ファイル | ミューテーション数 | Killed | Survived | スコア | 評価 |
|----------|-------------------|--------|----------|--------|------|
| calc.ts | 10 | 9 | 1 | 90% | Strong |
| validator.ts | 12 | 8 | 4 | 67% | Moderate |

## 未検出ミューテーション（テストギャップ）

### Critical（P1-P2 の未検出 — 対応必須）
- `src/domain/validator.ts:25` — `>=` → `>`: 境界条件がテストされていない
  - **推奨テスト**: 境界値でのテストケースを追加

### Warning（P3-P4 の未検出 — 対応推奨）
- `src/utils/calc.ts:32` — `+ 2` → `+ 3`: 数値計算がテストされていない
  - **推奨テスト**: 特定の入力値での計算結果を検証するテストを追加

### Info（P5 の未検出 — 参考情報）
- ...

## 推奨アクション
1. [最も重要なテストギャップの修正提案]
2. [次に重要な提案]
```

## スコア基準

| スコア | 評価 | 意味 |
|--------|------|------|
| > 90% | **Strong** | テストスイートは非常に強力 |
| 80-90% | **Good** | 良好だが改善の余地あり |
| 60-80% | **Moderate** | 重要なテストギャップがある |
| < 60% | **Weak** | テストカバレッジに深刻な問題 |

## 注意事項

- ミューテーションテストは計算コストが高い。`all` を指定する場合、長時間かかる可能性がある
- 各ファイルで5-15個のミューテーションを選択する（ファイルサイズに応じて調整）
- worktree 隔離環境内でのみファイル編集を行い、メインコードベースは変更されない
