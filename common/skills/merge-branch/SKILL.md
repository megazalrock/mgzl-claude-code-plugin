## コンテキスト

- 引数: `$ARGUMENTS`
- 現在のブランチ: !`git rev-parse --abbrev-ref HEAD`
- ワーキングツリーの状態: !`git status --porcelain`

## タスク

指定されたブランチを現在のブランチにマージする。コンフリクトが発生した場合は各ファイルを読み込んで文脈解析により自動解消する。

## ワークフロー

### Step 1: 引数の解析

`$ARGUMENTS` をスペース区切りでトークン化し、以下の通り分類する:

- `-y`: 確認スキップフラグ（位置不問、任意・1つ）
- それ以外のトークン: マージ元ブランチ名（必須・1つ）

ブランチ名が指定されていない場合は「マージするブランチ名を指定してください。」と表示して終了する。

### Step 2: 事前チェック

順次実施し、いずれかが失敗したら理由を明示して終了する。

1. **ワーキングツリーがクリーンか**
   - `git status --porcelain` の出力が空であることを確認
   - 空でなければ「ワーキングツリーに未コミットの変更があります。コミットまたはスタッシュしてから再実行してください。」と表示して終了
2. **指定ブランチがローカルに存在するか**
   - `git rev-parse --verify <branch>` が成功することを確認
   - 失敗した場合は「ブランチ '<branch>' がローカルに存在しません。」と表示して終了
3. **同一ブランチでないか**
   - 指定ブランチが現在のブランチと同じ場合は「指定されたブランチは現在のブランチと同じです。」と表示して終了

### Step 3: 確認

`-y` フラグが**含まれていない**場合のみ実施する。`-y` が含まれている場合はスキップして Step 4 へ進む。

1. 以下のメッセージを表示する（XXX = 指定ブランチ、YYY = 現在のブランチ）:
   ```
   XXXブランチをYYYブランチにマージします。
   ```
2. AskUserQuestion で続行確認を行う:
   - `question`: `マージを実行しますか？`
   - `header`: `マージ確認`
   - `multiSelect`: `false`
   - `options`（**順序固定・変更厳禁**）:
     1. `label: マージする` / `description: マージを実行する`
     2. `label: 中止する` / `description: マージを中止する`
3. 「中止する」が選択された場合は「マージを中止しました。」と表示して終了

### Step 4: マージ実行

`git merge <branch>` を実行する。終了コードと出力で以下のように分岐する:

- **成功 (Fast-forward / コンフリクトなし)**: マージ結果を表示し、Step 7 の結果報告へジャンプする
- **コンフリクト発生**: Step 5 へ進む
- **その他のエラー**: エラー出力を表示し、`git merge --abort` を提案して終了

### Step 5: コンフリクトの文脈解析・自動解消

1. **コンフリクトファイル一覧の取得**
   - `git diff --name-only --diff-filter=U` で取得
   - ユーザーに対象ファイル一覧を提示する

2. **ファイルごとに以下を実施**
   - Read でファイル全体を読む
   - `<<<<<<<` `=======` `>>>>>>>` マーカーで区切られた各コンフリクトブロックを特定する
   - 各ブロックについて、HEAD 側（現在のブランチ）と相手側（マージ元）の意図を比較して以下のいずれかを判断する:
     - **両方統合**: import 文の追加、配列要素の追加、独立したフィールド追加など、両方の変更が共存可能な場合は両方を残す
     - **片方採用**: 同じ箇所の異なる修正など、どちらかが正しい場合はその根拠を内部で記録した上で採用
     - **判断不能**: 文脈から決定できない場合はマーカーを残したまま該当ファイルをスキップし、後でユーザーに報告する
   - Edit でコンフリクトマーカーを削除し、解消後の内容に置き換える

3. **解消できたファイルをステージ**
   - 解消が完了したファイルそれぞれに対し `git add <file>` を実行

4. **解消不能ファイルがある場合**
   - 解消できなかったファイル一覧と各ファイルで判断不能だった理由を提示する
   - 「コンフリクトを自動解消できないファイルがあります。`git merge --abort` でマージを中止するか、手動で解消してください。」と表示して終了
   - この時点で勝手に `git merge --abort` は実行しない（ユーザーの判断を待つ）

### Step 6: マージコミット

すべてのコンフリクトが解消されてステージされた状態で `git commit --no-edit` を実行する。
これにより git のデフォルトマージメッセージ（`Merge branch 'XXX' into YYY` 形式）が採用される。

### Step 7: 結果報告

以下の形式で結果を報告する:

```
✅ マージ完了
- マージ元: <指定ブランチ>
- マージ先: <現在のブランチ>
- 解消したコンフリクト: <ファイル数>件（コンフリクトが発生した場合のみ）
  - <file1>: <解消方針の要約>
  - <file2>: <解消方針の要約>
```

コンフリクトが発生しなかった場合は「解消したコンフリクト」行は省略する。
