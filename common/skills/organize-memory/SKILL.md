## コンテキスト

- 引数: `$ARGUMENTS`
- プロジェクトルート: !`echo "${CLAUDE_PROJECT_DIR:-$(pwd)}"`
- 直近の変更: !`git log --oneline -n 20 2>/dev/null || echo "(git 履歴を取得できません)"`

## タスク

現在のプロジェクトの auto memory ディレクトリを走査し、各メモリーが現在のプロジェクト状況と乖離していないか、内容が冗長でないか、インデックス (`MEMORY.md`) と整合しているかを検査して整理する。

`-y` フラグが含まれる場合は、整理案の最終確認以外の確認ステップをスキップする。ただし、削除・統合の実行前には**必ず**ユーザー承認を取る。

## ワークフロー

### Step 1: メモリーディレクトリの特定

1. プロジェクトの絶対パスを取得する
   - `${CLAUDE_PROJECT_DIR}` が設定されていればそれを使用
   - 設定されていなければ `pwd` の出力を使用
2. パスを `~/.claude/projects/` 配下のキーに変換する
   - 絶対パス中の `/` をすべて `-` に置換する（先頭の `/` も `-` になる）
   - 例: `/Users/otto/workspace/mgzl-claude-code-plugin` → `-Users-otto-workspace-mgzl-claude-code-plugin`
3. メモリーディレクトリは `$HOME/.claude/projects/<変換後キー>/memory/` となる
4. ディレクトリの存在を `test -d` で確認する
   - 存在しない場合は「このプロジェクトには auto memory が未作成です: <path>」と表示して終了

### Step 2: メモリーの全件読み込み

1. メモリーディレクトリ配下を `ls` で一覧する
2. `MEMORY.md` を Read で読み込む（インデックス）
3. `MEMORY.md` 以外の `.md` ファイルを全て Read で読み込む
4. 各メモリーから以下を抽出する:
   - `name` (フロントマター)
   - `description` (フロントマター)
   - `type` (フロントマター直下 `type` または `metadata.type` のいずれか — 古い形式と新しい形式が混在しうる)
   - 本文（**Why:** / **How to apply:** などの構造)
   - ファイル更新日時（任意・参考程度）

### Step 3: プロジェクト現状の収集

検査の根拠とするため、以下の情報を取得する。

- 直近の git ログ（既にコンテキストに取得済み）
- メモリー内で言及されているファイル・関数・スキル・エージェント名がある場合、その存在を都度確認する（Read / Glob / Bash で `test -e`）

ここで一括で全部読む必要はない。Step 4 の検査時に必要なものだけ確認すれば十分。

### Step 4: 検査

各メモリーに対して、以下の3観点で評価する。

#### 観点A: 陳腐化（Staleness）

- メモリーが言及するファイル・関数・スキル・エージェント・コミットが**現在の状態で存在しないか改名されていないか**を確認する
- `project` 型メモリーで「××までに〇〇する」のような期限付き情報が**既に過去のものになっていないか**を確認する
- `reference` 型メモリーの参照先（外部URL以外）が**現存するか**を確認する
- `feedback` 型メモリーは原則として陳腐化しにくいが、参照しているプロジェクト固有の状態（特定ファイル等）が消えていれば該当する

#### 観点B: 冗長性（Redundancy）

- **同じ趣旨のメモリー**が複数ある場合、統合候補とする
- **より広い概念のメモリーに包含される**メモリーがある場合、被包含側を統合候補とする
- 統合する場合は **Why** と **How to apply** の文脈を失わないよう、双方の情報を残す

#### 観点C: インデックス整合性

- `MEMORY.md` に記載があるが**実体ファイルが存在しない**エントリ（壊れたリンク）
- 実体ファイルが存在するが**`MEMORY.md` に記載がない**エントリ（孤立ファイル）
- `MEMORY.md` の説明文がメモリー本体の `description` と**著しく乖離**しているエントリ

### Step 5: 整理案の作成

検査結果をもとに、以下のカテゴリで整理案をまとめる。**変更がないカテゴリは出力しない**。

```
## メモリー整理案

### 削除候補（陳腐化）
- <ファイル名>: <削除理由>

### 統合候補（冗長性）
- <統合後ファイル名> ← <統合元1>, <統合元2>: <統合理由>

### 内容更新候補（部分的な陳腐化）
- <ファイル名>: <更新が必要な箇所と理由>

### インデックス修正
- 壊れたリンク削除: <エントリ>
- 孤立ファイル追加: <ファイル名> — <description>
- 説明文の修正: <エントリ> — <新しい説明>
```

すべてのカテゴリで該当が無かった場合は「メモリーは整理済みです。変更はありません。」と表示して終了する。

### Step 6: 承認と確認

`-y` フラグの有無にかかわらず、整理案を AskUserQuestion で確認する。整理は不可逆な変更を含むため、自動承認は許容しない。

- `question`: `整理案を適用しますか？`
- `header`: `整理確認`
- `multiSelect`: `false`
- `options`（**順序固定**）:
  1. `label: 全て適用` / `description: 整理案を全て適用する`
  2. `label: 一部のみ適用` / `description: 適用する項目を個別に選択する`
  3. `label: 中止` / `description: 何も変更せず終了する`

「一部のみ適用」が選択された場合は、`multiSelect: true` の AskUserQuestion で適用したい項目を個別に選ばせる（選択肢は整理案の各項目）。

「中止」が選択された場合は「整理を中止しました。」と表示して終了。

### Step 7: 適用

承認された項目を以下の順で適用する。

1. **内容更新**: Edit で該当箇所のみ修正する
2. **統合**: 統合後ファイルを Write で作成（または既存に Edit で追記）し、統合元ファイルを `trash` で削除する
3. **削除**: 該当ファイルを `trash <path>` で削除する（`rm` は使用しない）
4. **インデックス修正**: `MEMORY.md` の該当行を Edit で削除・追加・修正する

各操作の後、変更内容を1行で報告する（例: `削除: feedback_xxx.md (理由)`）。

### Step 8: MEMORY.md の最終確認

すべての変更後、`MEMORY.md` を再度 Read で読み込み、以下を確認する:

- 残っているメモリーファイル全てが `MEMORY.md` に記載されているか
- `MEMORY.md` に記載されているエントリの実ファイルが全て存在するか
- 200 行を超えていないか（auto memory の仕様上、それ以降は切り捨てられる）

不整合があれば自動で修正する。200 行超過の場合はユーザーに警告のみ表示する（インデックスの圧縮は別タスク）。

### Step 9: 結果報告

以下の形式で結果を報告する:

```
✅ メモリー整理完了
- 対象ディレクトリ: <path>
- 削除: <件数>件
- 統合: <件数>件
- 内容更新: <件数>件
- インデックス修正: <件数>件
- 残ったメモリー数: <件数>件
```

変更が無い項目は省略する。

## 補足事項

### メモリーフロントマターの形式

新旧の形式が混在しうる。**両形式とも有効**として扱うこと。

- 新形式:
  ```yaml
  metadata:
    type: feedback
  ```
- 旧形式:
  ```yaml
  type: feedback
  ```

更新する際は、既存ファイルの形式を維持する（無理に統一しない）。

### 削除コマンド

このPCでは `rm` が `trash` のエイリアスになっているため、`/bin/rm` ではなく `trash <path>` を直接使用する。

### 慎重な判断

- 「陳腐化していそう」程度の曖昧な根拠で削除案に含めない。**確実に現状と矛盾している**ものだけを対象とする
- `feedback` 型は基本的に普遍的な指示なので、明確に状況が変わった証拠がない限り削除候補に含めない
- 統合は本当に同じ趣旨のもの同士のみ。「似ているけど観点が違う」ものは統合しない
