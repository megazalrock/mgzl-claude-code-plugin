# Memory Cleanup

現在のプロジェクトのメモリをスキャン・分析し、不要なメモリを積極的に削除する。メモリは少ないほど良い — 本当に価値のある非自明な知見だけを残す。

分析処理は `@memory-analyzer` エージェントに委譲し、カテゴリ単位で並列実行する。

## 対象メモリ

### プロジェクトメモリ
- **場所**: 現在のプロジェクトに対応する `~/.claude/projects/<project>/memory/`
- **構造**: `MEMORY.md`（インデックス）+ 個別 `.md` ファイル（YAML frontmatter: name, description, type）

### サブエージェントメモリ
- **場所**: 現在のワークスペース配下の `.claude/agent-memory-local/<agent-name>/`
- **構造**: `MEMORY.md`（エージェント固有の知識を含むことがある）+ 個別 `.md` ファイル
- サブエージェントの MEMORY.md はインデックスだけでなく、コンベンション情報やプロジェクト固有知識を直接含む場合がある

## 実行フロー

### Step 1: メモリのスキャン

#### 1a. プロジェクトメモリの探索

現在のプロジェクトに対応するメモリディレクトリを特定する。パスは `~/.claude/projects/` 配下にあり、ワークスペースの絶対パスをハイフン区切りに変換した名前のディレクトリになる（例: `/Users/otto/workspace/myapp` → `-Users-otto-workspace-myapp`）。

```
Glob: ~/.claude/projects/<current-project>/memory/*.md
```

#### 1b. サブエージェントメモリの探索

サブエージェントのメモリは、エージェント定義ファイルを元に現在のワークスペース内を探索する。

1. **エージェント名の収集**: インストール済みプラグインのエージェント定義ファイルを探索する:
   ```
   Glob: ~/.claude/plugins/**/agents/*.md
   ```
   各ファイルのパスからエージェント名を導出する。メモリディレクトリ名は `<plugin名>-<agent名>` の形式になる。
   例: `cbo/agents/browser-operator.md` → `cbo-browser-operator`

2. **メモリディレクトリの探索**: 現在のワークスペースに対して、収集したエージェント名でメモリディレクトリを確認する:
   ```
   <current-workspace>/.claude/agent-memory-local/<agent-name>/
   ```
   ディレクトリが存在する場合、配下の全 `.md` ファイルを収集する。

#### 1c. グルーピングとサマリー表示

発見したファイルをカテゴリごとにグルーピングする:
- プロジェクトメモリ: 1グループ（現在のプロジェクト）
- サブエージェントメモリ: エージェント単位（例: `cbo-code-fix-executor`, `cbo-implementation-plan-reviewer`）

以下の形式でサマリーテーブルを表示する:

```
## スキャン結果

### プロジェクトメモリ
| メモリ数 | ファイル |
|---------|---------|
| 3       | feedback_xxx.md, ... |

### サブエージェントメモリ
| エージェント | メモリ数 | ファイル |
|------------|---------|---------|
| cbo-code-fix-executor | 2 | MEMORY.md, feedback_xxx.md |
```

### Step 2: エージェントによる並列分析

カテゴリごとに `@memory-analyzer` エージェントを起動し、並列で分析を実行する。

各エージェントへの指示:
```
以下のメモリファイルを分析し、クリーンアップレポートを JSON 形式で出力してください。

## カテゴリ
{プロジェクト名 or エージェント名}

## MEMORY.md
{MEMORY.md のパス（存在する場合）}

## メモリファイル
{対象ファイルパスの一覧}

## 参考: CLAUDE.md の内容
{該当プロジェクトの CLAUDE.md があれば抜粋}
```

カテゴリ数が少ない場合（3つ以下）はすべて並列起動する。4つ以上の場合は3つずつバッチ実行する。

### Step 3: レポートの統合と提示

全エージェントの結果（JSON）を集約し、以下の形式で統合レポートを表示する:

```markdown
# Memory Cleanup Report

## サマリー
- スキャン対象: N プロジェクト、M エージェント、L メモリ
- 削除推奨: X 件
- 修正推奨: Y 件
- 問題なし: Z 件

## 削除推奨

### 1. [カテゴリ名] feedback_example.md
- **理由**: 言及しているバグ #27881 は修正済み
- **内容抜粋**: > "worktree の isolation バグにより..."

### 2. [カテゴリ名] MEMORY.md
- **理由**: 内容が CLAUDE.md と重複
- **内容抜粋**: > "テストは Vitest を使用..."

## 修正推奨

### 1. [カテゴリ名] MEMORY.md
- **理由**: feedback_xxx.md がインデックスに未登録
- **修正内容**: エントリを追加

## 問題なし
- [カテゴリ名] feedback_no_auto_commit.md — ユーザー固有の強い好み、引き続き有効
```

### Step 4: ユーザー承認

AskUserQuestion で実行方法を選択させる:
- **すべて実行**: 推奨されたすべてのアクション（削除・修正）を一括実行
- **選択して実行**: 各推奨を1つずつ確認し、実行するかスキップするか選択
- **キャンセル**: 何もせずに終了

### Step 5: クリーンアップの実行

承認された各アクションを実行する:

1. **ファイル削除**: 対象ファイルを削除し、対応する MEMORY.md からエントリを除去する
2. **インデックス修正**: MEMORY.md にエントリを追加/削除する
3. **品質修正**: 個別メモリファイルの frontmatter や構造を修正する
4. **MEMORY.md が空になった場合**: MEMORY.md 自体も削除する
5. **サブエージェントディレクトリが空になった場合**: ディレクトリも削除する

### Step 6: 結果報告

```
## 実行結果
- 削除: N ファイル
- 修正: M ファイル
- 残存メモリ: K 件（P プロジェクト、Q エージェント）
```

## 注意事項

- `~/.claude/projects/` 配下や `.claude/agent-memory-local/` 配下のファイル操作はサンドボックス制限に抵触する可能性がある。権限ダイアログが表示された場合は許可が必要
- 削除は不可逆。Step 3 のレポートで必ず内容を確認してから実行する
- セッションファイル（`.jsonl`）やメモリ以外のファイルには一切触れない
