# コード実装の知見蓄積手順

コードレビュー完了後、レビューで検出された指摘から**汎用的なコード実装時の教訓**を抽出し蓄積するための共通手順。`review:diff` / `review:file` / `impl:execute` から参照される。

## 前提

- 呼び出し元は、統合済みのレビュー結果と、そこに含まれる `[3]` 推奨以上（`[3]`/`[4]`/`[5]`）の指摘件数を把握していること。

## 発火条件

- 統合レビュー結果に `[3]` 推奨以上の指摘が **1 件以上** ある場合のみ実行する。
- `[2]` 以下のみ、または指摘 0 件の場合は蓄積をスキップする。

## 手順

1. 教訓ファイルのパス: `$MGZL_DIR/knowledge/implementation-lessons.md`
   - これは実装計画書作成の教訓 (`implementation-plan-lessons.md`) とは別物。コードを実装する際の知見を貯める。
2. ファイルもディレクトリも存在しない場合は、ディレクトリを作成し、このファイルと同じディレクトリにある [テンプレート](lessons-template.md) を読み込んで新規作成する。
3. レビューで指摘された内容を振り返り、**汎用的な教訓**を抽出する。
4. **汎用的な教訓の判定基準**（以下の全てを満たすもののみ記録する）:
   - 今回のプロジェクト・機能・ファイル名を削除しても意味が通じる
   - 他のコード実装でも再発・適用しうる
   - 「〇〇の場合は△△すべき」のように条件と対策のペアで表現できる
   - **記録しない例**: 「UserList.vue の props が any になっている」（特定ファイル依存）
   - **記録する例**: 「外部入力をテンプレートに直接展開する場合は XSS を考慮しエスケープ／サニタイズすべき」（汎用）
5. **エントリの記述フォーマット**: `- YYYY-MM-DD: [条件・状況] → [教訓・対策]`
6. レビュー観点に対応するカテゴリセクションの末尾に追記する（カテゴリは教訓ファイルのセクション見出しに従う）。

   | レビュー観点（サブエージェント） | カテゴリ見出し |
   |------|------|
   | reviewer-for-style | `## コードスタイル・命名・配置` |
   | reviewer-for-logic | `## ロジック・エッジケース・例外処理` |
   | reviewer-for-design | `## 設計・責務分離・原則違反` |
   | reviewer-for-security-performance | `## セキュリティ・パフォーマンス` |
   | reviewer-for-comments | `## コメント品質` |
   | reviewer-for-test-code | `## テストコード品質` |
   | 上記に分類できないもの | `## その他` |

7. **重複チェック**: 既存の教訓と実質的に同じ内容の場合は追記しない。
8. **統合**: カテゴリ内の教訓が 5 件を超えた場合、類似の教訓を 1 件に統合して件数を減らす。
