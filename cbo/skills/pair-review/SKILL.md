# ペアプロ用軽量レビュー

ペアプログラミング中に、作業中のコード変更に対して**本当に重要な問題だけ**を素早く指摘するスキル。
開発の流れを止めないことが最優先。ノイズになる些末な指摘は絶対にしない。

**このスキルの実行ではファイルの修正を行ってはならない**

またこのスキルは `/loop` コマンドで実行されることを想定しているので、
**`&&` などでコマンドを結合し、許可された形式以外の方法でコマンドを実行してはならない**


## 実行手順

### 1. 変更ファイルの取得

以下の2つのコマンドで変更内容を取得する:
```
git diff HEAD --name-only
git diff --staged --name-only
```

両方の結果を統合し、重複を除いたファイル一覧を作成する。
変更がない場合は「変更ファイルがありません」と伝えて終了。

### 2. 差分の取得と読み込み

変更ファイルごとに差分を取得する:

```
git diff HEAD -- <filepath>
git diff --staged -- <filepath>
```

差分だけでは文脈が不足する場合、変更されたファイル全体を読む。
ただし、変更に関係しない部分まで深追いしないこと。

### 3. プロジェクトルールの確認

CLAUDE.md、CLAUDE.local.md、.claude/rules/ の内容を把握した上でレビューを行う。
これらは会話コンテキストに既に含まれているため、改めて読む必要はない。

### 4. レビュー

変更内容を深く分析する。

#### 報告すべき問題（これだけを報告する）

- **重大なバグ**: nullポインタ参照、無限ループ、データ消失、競合状態、off-by-oneエラーなど、実行時に障害を引き起こす問題
- **セキュリティ脆弱性**: XSS、インジェクション、認証バイパス、機密情報の漏洩、OWASP Top 10に該当する問題
- **コーディング原則違反**: プロジェクトの CLAUDE.md / .claude/rules/ に明記されたルールへの違反
  - 例: プロジェクト固有の禁止事項、`!` / `as` / `any` の不必要な使用
- **よりよいコードの改善提案**: より効率的なアルゴリズム、実装の単純化、メンテナンスビリティの向上などができる場合に、確実に適用可能な提案を行う
  - コードの改善提案は実装が現実的な工数で可能なものである必要がある
  - 関連する実装や型と矛盾がないかよく調査した上で提案を行う

#### 報告してはいけないもの（これらは無視する）

- TypeScriptの型エラーや型の不整合（型チェックツールの仕事）
- ESLint / Prettierで検出できるコードスタイル違反
- 命名規則の好み（camelCase vs snake_case など）
- コメントやドキュメントの不足
- importの順序
- テストの不足（テストレビューは別スキルの仕事）

### 5. 結果の報告

#### 問題が見つかった場合

問題がなかった点、どの様に検証したかなどは報告に含めない

ファイルパスと行番号を明記して、簡潔に報告する:

```
## 発見された問題

### [重大度] ファイル名:行番号
問題の説明（1-2文）
該当コードの抜粋（必要な場合のみ）
```

重大度は以下のいずれか:
- **BUG**: 実行時に障害を引き起こすバグ
- **SECURITY**: セキュリティ脆弱性
- **RULE**: プロジェクトルール違反

#### 問題が見つからなかった場合

「重大な問題は見つかりませんでした」と一言だけ伝えて終了する。
長々と「レビューしました、以下の観点でチェックしました...」のような報告は不要。
