## コンテキスト

- 対象スキル: $ARGUMENTS

## ワークフロー

### Step 1: 対象スキルの特定

- 「対象スキル」が指定されている場合は `.claude/skills/` 配下から該当スキルを特定
- 指定がない場合は AskUserQuestion で対象スキルを選択させる
- 対象が見つからない場合はその旨を伝え終了

### Step 2: 現状分析

対象スキルの以下を読み取る：
- `SKILL.md`（フロントマター + 本文）
- ディレクトリ構造（`resource/`, `scripts/`, `reference/` 等）
- 関連ファイル（参照されているファイル）

### Step 3: 分析基準の取得

以下の参照ファイルを読み込む：
- [ベストプラクティス](references/skill-authoring-best-practices.md) - 一般的なスキル設計の基準

### Step 4: 分析レポートの作成

チェックリストに基づき、改善ポイントを特定してレポートを作成する。

**チェックリスト:**

| カテゴリ | チェック項目 |
|---------|-------------|
| フロントマター | `name` がケバブケースまたはグループ:名前形式であるか |
| フロントマター | `description` が日本語でトリガーワードを含んでいるか |
| フロントマター | `argument-hint`, `allowed-tools`, `disable-model-invocation` が適切に設定されているか |
| 簡潔さ | SKILL.md が500行以内に収まっているか |
| 簡潔さ | Claudeが既に知っている情報を冗長に説明していないか |
| 構造 | ワークフローが明確なステップで構成されているか |
| 構造 | Progressive disclosure が適切か（詳細は参照ファイルに委譲） |
| 構造 | ファイル参照の深さが1階層以内か |
| 自由度 | タスク特性に応じた適切な自由度が設定されているか |
| 用語 | プロジェクトの用語集と一貫しているか |
| ガイドライン | プロジェクト固有ガイドラインに準拠しているか |

各項目について OK / 要改善 / 該当なし を判定し、要改善の項目には具体的な改善案を添える。

### Step 5: ユーザー確認

分析レポートを提示し、AskUserQuestion でリファクタリングを実行するか確認する。
- 実行する場合は Step 6 へ
- 実行しない場合は終了

### Step 6: リファクタリング実行

承認された改善を対象スキルのファイルに適用する。

**適用時の注意:**
- 変更は最小限に留め、既存の動作を壊さない
- フロントマターの変更は正確にYAML構文を守る
- 参照ファイルの変更時は、SKILL.md内のリンクとの整合性を確認する
- 大幅な構造変更がある場合は、変更前後の差分を明示する

### Step 7: 結果報告

変更内容のサマリーを以下の形式で提示する：
- 変更したファイル一覧
- 各ファイルの変更概要
- 改善前後の比較（主要な変更点のみ）
