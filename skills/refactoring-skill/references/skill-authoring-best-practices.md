# Skill設計ベストプラクティス（サマリー）

Anthropic公式ドキュメントから、レビュー時に特に重要なポイントを抽出したもの。

## 核心原則

### 1. 簡潔さが最重要
- コンテキストウィンドウは共有リソース。SKILL.mdに含める情報は最小限に
- Claudeが既に知っている情報（一般的な概念、ライブラリの基本的な使い方等）は書かない
- SKILL.md本文は500行以内を推奨

### 2. 適切な自由度の設定
- **高い自由度**: 複数のアプローチが有効な場合（コードレビュー等）
- **中程度の自由度**: 推奨パターンがあるが多少の変動を許容する場合
- **低い自由度**: 操作が脆弱でエラーが起きやすい場合（DB操作、特定コマンド実行等）

### 3. Progressive Disclosure
- SKILL.mdは目次のように、詳細は別ファイルに委譲する
- 参照ファイルの深さは1階層以内に留める（参照先から更に参照を辿らない）
- 100行を超える参照ファイルには目次をつける

## チェックリスト

### フロントマター
- [ ] `name`: 64文字以内、小文字/数字/ハイフンのみ
- [ ] `description`: 1024文字以内、何をするか＋いつ使うかを明記
- [ ] descriptionは三人称で記述（「I can...」「You can...」は避ける）

### 内容品質
- [ ] SKILL.md本文が500行以内
- [ ] Claudeが既知の情報を冗長に説明していない
- [ ] 時間依存の情報が含まれていない
- [ ] 用語が一貫している
- [ ] 具体的な例がある（抽象的でない）
- [ ] ファイル参照が1階層以内

### ワークフロー
- [ ] 複雑なタスクは明確なステップに分解されている
- [ ] フィードバックループが含まれている（検証→修正→再検証）
- [ ] 条件分岐がある場合は明示されている

### スクリプト（該当する場合）
- [ ] エラーをClaude任せにせず、スクリプト内で処理している
- [ ] マジックナンバーがない（値の理由がコメントされている）
- [ ] 必要なパッケージが明記されている

## アンチパターン
- Windowsスタイルのパス（`\`）を使わない
- 選択肢を多く提示しすぎない（デフォルトを示し、代替はエスケープハッチとして提供）
- 深いネストの参照を避ける
