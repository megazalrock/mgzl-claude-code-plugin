## コンテキスト

- ワーキングツリーの状態： !`git status`

## タスク

`$ARGUMENTS` を解析し、以下のフラグの有無に基づいて処理を分岐する。

- `-y`: コミットメッセージを自動選択する（ユーザーへの確認をスキップ）
- `--staged`: ステージ済みの差分のみを対象とする
- 両方のフラグは組み合わせ可能（例: `-y --staged`、`--staged -y`）

### `--staged` が含まれる場合（ステージ済みのみコミット）

1. ステージされた差分が無い場合はその旨をユーザーに通知し終了
2. `git diff --staged` でステージ済み差分を取得・分析して、コミットメッセージの生成ルールに従いコミットメッセージを3パターン生成する
3. コミットメッセージの選択:
   - `-y` が含まれる場合: 自動選択ルールに従い最適な1つを自動選択し、選択したメッセージとその理由をユーザーに表示する
   - `-y` が含まれない場合: ユーザーに生成した3パターンのコミットメッセージを AskUserQuestion で選択させる
4. ステージされた差分をコミットする（※コミットコマンドの形式ルールに従うこと）

### `--staged` が含まれない場合（全変更をコミット）

1. 現在のワークスペースがクリーンな場合はユーザーに通知して終了する
2. `git diff` と `git diff --staged` で全ての差分を取得・分析して、コミットメッセージの生成ルールに従いコミットメッセージを3パターン生成する
3. コミットメッセージの選択:
   - `-y` が含まれる場合: 自動選択ルールに従い最適な1つを自動選択し、選択したメッセージとその理由をユーザーに表示する
   - `-y` が含まれない場合: ユーザーに生成した3パターンのコミットメッセージを AskUserQuestion で選択させる
4. `git add -A` で全ての変更をステージする
5. ステージされた差分をコミットする（※コミットコマンドの形式ルールに従うこと）

## コミットコマンドの形式ルール

- コミットは必ず `git commit -m "メッセージ"` のシンプルな形式で実行すること
- HEREDOC（`cat <<'EOF'`）や `$()` コマンド置換は絶対に使用しないこと
- これは `allowed-tools` のパターンマッチに正しく合致させるために必須である

## コミットメッセージ生成ルール

- 変更されたコード量で判断せず、何が変更の主題かをよく分析する
- 実装の追加・修正、テストの追加・修正、リファクタリング、の優先順位でコミットメッセージに含める
- 先頭にConventional Commitsのtypeを追加する
  - Scopeは追加しない
- 1行で記述する
- プレーンテキストで記述する
- 変数名や関数名などはバッククォートで囲う

### 適切なコミットメッセージの例
```
feat: その他の予定に協力会社をアサインできる機能を追加
fix: ホワイトボードのレイアウト崩れを修正
refactor: useCommonStore.tsのテストをリファクタ
refactor: `useGanttMonthlyStore`にTSDocを追加
chore: コメントのtypoを修正
```

## 自動選択ルール（`-y` 指定時）

3つの候補から最適なコミットメッセージを1つ選択する。以下の基準を優先度順に適用する:

1. **正確性**: 変更の主題を最も正確に表現しているメッセージを優先する
2. **簡潔性**: 同程度に正確であれば、より簡潔なメッセージを優先する
3. **typeの適切さ**: Conventional Commitsのtypeが変更内容に最も合致しているメッセージを優先する

### 自動選択時の出力形式

選択後、コミット実行前に以下の形式でユーザーに表示する:

```
自動選択されたコミットメッセージ:
> {選択されたメッセージ}

選択理由: {なぜこのメッセージが最適と判断したかの1文の説明}
```

### AskUserQuestionの選択肢で表示する内容

- `label`: コミットメッセージ
- `description`: コミットメッセージで何を重視したかの簡潔な説明
