# update-implementation-step-executor

このリポジトリ (`mgzl-claude-code-plugin`) の開発時専用スキル。
プロジェクト側で蓄積された**コード実装の教訓ナレッジ**を精査し、**汎用的な知見だけ**を `cbo/agents/implementation-step-executor.md` に織り込む。

教訓を「ランタイム参照されるナレッジファイル」に置きっぱなしにすると、サブエージェント呼び出し時に追加読込が必要になりプロンプト密度も下がる。本スキルは "蒸留して本体に取り込む" 流儀でサブエージェントの初期能力を高める方針。

## 入力

- `$1` — 教訓ナレッジファイルの絶対パス（任意）
  - 省略時のデフォルト: `/Users/otto/workspace/craftbank/arrangement-env/front/.mgzl/knowledge/implementation-lessons.md`
- 反映先（固定）: `/Users/otto/workspace/mgzl-claude-code-plugin/cbo/agents/implementation-step-executor.md`

## ワークフロー

### ステップ1: 入力ファイルの読込

1. ナレッジファイルパスを確定する（`$1` があれば優先、無ければデフォルト絶対パス）。
2. ナレッジファイルが存在しない場合はユーザーに通知して終了する（勝手に別ファイルを探さない）。
3. ナレッジファイル全文を `Read` で取り込む。
4. 反映先 `cbo/agents/implementation-step-executor.md` を `Read` で取り込む。

### ステップ2: 教訓の精査と汎用化判定

各教訓に対して以下のチェックを行う：

**汎用化の可否**
- そのまま採用 — 言語・フレームワーク・テスト基盤の一般原則（TypeScript 厳格設定、外部入力のエスケープ／サニタイズ、N+1 回避、例外時の早期 return 等）。
- 抽象化して採用 — 特定の Store 名・コンポーネント名・ファイル名などプロジェクト固有要素を含む場合は、固有名を一般語（「state を保持する Store」「一覧表示の UI」等）に置き換えて採用。
- 不採用 — そのプロジェクト固有のドメイン仕様・特定ファイルのレビュー指摘で抽象化困難なもの。

**反映の必要性**
- 反映先エージェント定義を `Grep` などで検索し、同等の指針が既に書かれているなら "反映済み" として除外する。
- 表現が近いが視点が異なる場合は "補強候補" として残す。

### ステップ3: 反映先セクションの決定

`implementation-step-executor.md` の既存構造に合わせ、教訓を以下のいずれかに振り分ける。教訓ファイルのカテゴリ見出しと反映先セクションの対応は次のとおり：

| 教訓カテゴリ（lessons の見出し） | 反映先候補セクション |
|----------------------------------|----------------------|
| コードスタイル・命名・配置 | 「あなたの責務 > 2. 単一ステップの実装」 / 「実装プロセス > 2. 実装の実行」 |
| ロジック・エッジケース・例外処理 | 「あなたの責務 > 2. 単一ステップの実装」 / 「あなたの責務 > 4. 品質保証」 |
| 設計・責務分離・原則違反 | 「あなたの責務 > 2. 単一ステップの実装」 |
| セキュリティ・パフォーマンス | 「あなたの責務 > 4. 品質保証」 / 「実装プロセス > 4. コード解析」 |
| コメント品質 | 「あなたの責務 > 2. 単一ステップの実装」 |
| テストコード品質 | 「実装プロセス > 5. テストの実行」 / 「あなたの責務 > 4. 品質保証」 |
| その他 | 内容に最も近いセクション、または「注意事項」 / 新規サブセクション |

既存項目に統合できる場合は **新規箇条書きを増やすより既存文の補強を優先** する。膨張を避けるため。

### ステップ4: 反映案の提示

ユーザーには以下のフォーマットで列挙する。一覧性を最優先し、長文の引用は避ける：

```
[N件の汎用知見を抽出しました]

# 反映案
1. [カテゴリ] 教訓1行サマリ
   反映先: <セクション名>
   操作: 追記 / 既存文の補強 / 新規サブセクション
   差分プレビュー（±3行程度）

2. ...

# スキップ理由（参考）
- 「特定 Store の更新順依存」→ プロジェクト固有度が高く抽象化困難
- 「○○画面のバリデーション仕様」→ ドメイン固有
```

### ステップ5: ユーザー承認

`AskUserQuestion` を使い、以下の選択肢を提示する：

- すべて反映 — 提示した全項目を Edit で適用
- 項目を選択して反映 — 番号で取捨選択（番号リストを別途回答）
- 文言を調整したい — どの項目をどう書き換えるかをユーザーが指定
- 今回は反映しない — 何も変更せず終了

`AskUserQuestion` で得られなかった微調整は通常のテキスト応答で確認する。

### ステップ6: 反映実行

1. 承認された項目のみを `Edit` で `cbo/agents/implementation-step-executor.md` に反映する。
2. 1項目ずつ Edit を実行し、各 Edit の `old_string` は前後の文脈を 2〜3行含めて一意性を確保する。
3. 反映先のフロントマター（`name`, `description`, `model`, `tools`, `skills`, `memory` 等）は変更しない。
4. 全 Edit 完了後、`Read` で反映結果を再確認し、構造崩れ（重複、見出し階層ズレ）が無いか目視チェックする。

### ステップ7: 結果報告

以下の形式で報告する：

```
implementation-step-executor.md を更新しました

反映先: cbo/agents/implementation-step-executor.md
反映元: <ナレッジパス>
反映項目数: N件
スキップ項目数: M件（理由は前段で提示済み）

# 反映した知見の見出し
- ...
```

ナレッジファイル側は **編集しない**。蓄積ルール上、ナレッジ側のローテーションは別タスクで行う。

## 重要な制約

- **エージェント本体に織り込む**: ナレッジを単に再掲する `references/` を増やしたり、ランタイムで読み込ませたりはしない。教訓はサブエージェントの初期プロンプトに溶かし込む。
- **特定固有名は抽象化**: Store 名・コンポーネント名・ファイル名・関数名など特定プロジェクトのシンボルをそのまま記載しない。
- **既存記述との重複回避**: 既に書かれている指針と同じことを別言語で言わない。補強の場合は既存箇所を編集する。
- **フロントマター不可侵**: `tools`, `skills`, `memory` 等のメタ情報は本スキルで触らない。
- **承認なき反映は禁止**: `AskUserQuestion` で明示承認を得るまで `Edit` を呼ばない。
